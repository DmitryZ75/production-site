Standalone Bank

The primary objective of this project was to prepare the bank for sale. To achieve this, we needed to transform a highly integrated foreign bank into a fully autonomous entity without interrupting ongoing operations. Over 80% of the project was infrastructure-focused, making it both challenging and uniquely engaging.

In reality, there are no "foreign" banks in Russia per se—there are Russian banks owned by foreign entities, including major global banks. Operationally, these banks function with relative independence under Russian law. However, in terms of IT, integration can vary widely, from minimal to nearly complete. In our case, the level of integration was significant, meaning we had to find, create, or implement alternatives for each service. Despite the separation, the bank remained part of a global group, so we had to uphold stringent security and reliability standards, now entirely managed by the local team.

This gave rise to a series of infrastructure tasks to be completed within three months. Due to limited resources—primarily human—and time constraints, we opted for a phased migration to a new Active Directory (AD) domain built on the existing infrastructure, rather than constructing a fully parallel setup.

Project Scope

The scale of the project was immediately clear: roughly 150 servers, 300 user accounts, 400 workstations, and 400 service accounts were housed within the original AD's organizational units.

Project Phases

Phase 1: New Domain Creation

Our first step involved creating a new domain, deploying necessary services (AD, DNS, DHCP, CA, etc.), and creatively reimagining existing Group Policies to align with the new setup.

Phase 2: Establishing Trusted Relationships

We then established trusted relationships between the new domain and the main domain of the global bank, which included more than a dozen domains. Here, we encountered our first major challenge: the original domain was running Windows 2000, with all the associated limitations. Following extensive security checks by the information security team, we successfully established the necessary trusted relationships.

Phase 3: Workstation Migration

This stage involved migrating workstations and took around three months. The main constraint was the inability to create transitive trust relationships with the other domains in the global bank due to security policies. Some employees continued to work on workstations within the original domain until the separation date.

Phase 4: Server Transfer

Conducted in parallel with the third, this phase involved transferring servers from the original domain to the new one, a process that proceeded with minimal issues.

Phase 5: User and Service Object Migration

Finally, we migrated user and service objects (accounts and groups). This phase was marked by challenges with the Active Directory Migration Tool (ADMT) compatibility across both the original and new domains, as well as with Windows Server version requirements. Ultimately, we identified a viable solution using ADMT 3.1 on Windows Server 2008. Fortunately, password migration wasn’t a requirement, and we periodically ran object copying and updates until we froze changes in the original domain.

Conclusion

This project exemplified the complexities of untangling global integration while meeting strict timelines and high security standards—transforming an integrated foreign-owned bank into an independent entity ready for sale.
